# GAD-7 AI 辅助评估系统 — 项目目标与规格说明

## 1. 项目愿景与范围

### 1.1 目标

构建一套**前后端分离**的系统：用户通过**自然语言对话**完成 **GAD-7（广泛性焦虑障碍 7 项量表）** 的引导式访谈；服务端在对话流中**识别每一项对应的0–3 分**，在信息足够时**汇总总分与严重度解读**，并可选生成结构化记录。后端核心推理能力来自**团队自主训练/部署的 AI 模型**（可与规则引擎、量表引擎协同）。

### 1.2 非目标（本阶段）

- 前端视觉稿、组件库选型、无障碍细节由产品/设计**另行产出**；本文档仅约定**前端需依赖的契约与交互能力**。
- 不替代临床诊断；输出需带**免责声明**与**使用场景说明**（教育/筛查/辅助，非医疗结论）。

### 1.3 GAD-7 量表摘要（实现约束）

| 题号 | 维度说明（简述） | 单项分值 |
|------|------------------|----------|
| Q1–Q7 | 过去两周内「紧张、担忧、无法停止担心、难以放松、坐立不安、易怒、害怕」等频率 | 每项0–3 |
| **总分** | 七项相加 | **0–21** |

**严重度参考（可与产品文案统一）**

- 0–4：最小焦虑  
- 5–9：轻度  
- 10–14：中度  
- 15–21：重度  

（具体阈值与文案以临床/教研要求为准，系统应可配置。）

## 2. 系统架构概览

```
 HTTPS / WSS
  +----------------+       +-----------------------------+
  | 前端应用       | ----> | API 网关 / BFF              |
  | （另行设计）   |       | 鉴权、限流、会话、审计入口 |
  +----------------+       +-----------------------------+
                                      |
          +---------------------------+---------------------------+
          |                           |                           |
  +---------------+           +---------------+           +---------------+
  | 对话编排服务  | <-------> | 自研模型推理  | <-------> | 量表/打分引擎 |
  | 状态机 / RAG  |           | GPU / 推理 API |           | GAD-7 规则    |
  +---------------+           +---------------+           +---------------+
          |                           |
          v                           v
  +---------------+           +---------------+
  | 会话与消息存储 |           | 模型版本注册   |
  +---------------+           +---------------+
```

**原则**

- **自研模型**：负责自然语言理解与生成（追问、澄清、共情性引导）；**写入量表的最终分项分数**须经**确定性打分工具**校验与锁定，避免仅依赖自由文本推断（见第 4 节）。
- **打分工具**：对「每一项是否已可打分、分值多少、是否需澄清」给出**可解释、可审计**的结果。
- **编排层**：维护 GAD-7 访谈状态（当前题、缺失项、是否可出总结），并决定调用模型或仅走规则。

## 3. 接口安排（API 契约）

以下路径与字段为**目标契约**；实现时可加版本前缀（如 `/v1`）。

### 3.1 通用约定

- **认证**：`Authorization: Bearer <token>`（或会话 Cookie，二选一，文档化即可）。
- **内容类型**：`application/json`；流式对话使用 **SSE** 或 **WebSocket**（二选一或并存，建议 SSE 优先降低运维复杂度）。
- **错误体**（示例）：

```json
{
  "error": {
    "code": "INVALID_SESSION",
    "message": "会话不存在或已过期",
    "details": {}
  }
}
```

- **幂等**：创建会话、提交用户消息建议带 `Idempotency-Key` 头（可选）。

### 3.2 会话生命周期

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/sessions` | 创建评估会话；返回 `session_id`、初始系统提示/首条 AI 话术（若由服务端生成）。 |
| `GET` | `/sessions/{session_id}` | 拉取会话元数据：状态、`model_id`、量表进度摘要。 |
| `DELETE` | `/sessions/{session_id}` | 结束并归档（软删除）；可选匿名化。 |

**`POST /sessions` 请求体（示例）**

```json
{
  "locale": "zh-CN",
  "user_pseudo_id": "optional-stable-id",
  "preferences": {
    "tone": "supportive",
    "disclaimer_acknowledged": true
  }
}
```

**响应（示例）**

```json
{
  "session_id": "sess_xxx",
  "created_at": "2026-04-11T12:00:00Z",
  "state": "collecting",
  "gad7_progress": {
    "answered_items": [],
    "pending_items": ["Q1", "Q2", "Q3", "Q4", "Q5", "Q6", "Q7"]
  }
}
```

### 3.3 对话消息

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/sessions/{session_id}/messages` | 用户发送一条文本；服务端异步生成 AI 回复（非流式）或返回 `job_id`。 |
| `GET` | `/sessions/{session_id}/messages` | 分页拉取历史（审计、刷新页面）。 |

**`POST .../messages` 请求体**

```json
{
  "role": "user",
  "content": "我最近一周几乎每天都会担心很多事情。",
  "client_message_id": "uuid-for-dedup"
}
```

**响应（非流式，示例）**

```json
{
  "messages": [
    {
      "id": "msg_1",
      "role": "assistant",
      "content": "…",
      "created_at": "…"
    }
  ],
  "gad7_snapshot": {
    "items": {
      "Q1": { "score": null, "confidence": 0.0, "needs_clarification": true },
      "Q2": { "score": 2, "confidence": 0.86, "needs_clarification": false }
    },
    "total": null,
    "severity_band": null,
    "ready_for_summary": false
  }
}
```

### 3.4 流式对话（推荐）

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/sessions/{session_id}/messages:stream` | **SSE**：事件流推送 token/片段与最终 `gad7_snapshot`。 |

**SSE 事件类型（建议）**

- `token`：增量文本（可选）
- `message_complete`：完整 assistant 消息
- `gad7_update`：打分快照更新
- `done`：结束
- `error`：流内错误

### 3.5 打分与总结（显式操作，可选）

用于调试、人工复核或与第三方系统对接。

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/sessions/{session_id}/gad7/score` | 根据当前完整对话**重新运行**打分管道（不新增用户消息）。 |
| `GET` | `/sessions/{session_id}/gad7/report` | 返回结构化报告：每项分数、依据片段引用、总分、严重度、免责声明文本。 |

**`GET .../report` 响应（示例）**

```json
{
  "questionnaire": "GAD-7",
  "items": [
    {
      "id": "Q1",
      "prompt_ref": "official_zh_short",
      "score": 2,
      "evidence_spans": [{ "message_id": "msg_3", "start": 12, "end": 45 }],
      "rationale": "用户表述近两周多数日子存在紧张感，对应「超过一半天数」。"
    }
  ],
  "total": 12,
  "severity_band": "moderate",
  "summary_text": "…",
  "disclaimer": "本结果仅供筛查与自助参考，不能替代专业诊断。"
}
```

### 3.6 模型与版本管理（运维/研发）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/admin/models` | 列出已注册自研模型：`model_id`、部署地址、能力标签。 |
| `POST` | `/admin/models/{model_id}/rollout` | 灰度比例、回滚（需管理员权限）。 |

面向前端的**推理模型选择**可通过创建会话时指定或后端默认策略；前端仅需展示当前 `model_id`（可选）。

### 3.7 健康检查与观测

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/health` | 进程存活 |
| `GET` | `/health/ready` | 依赖就绪（DB、模型网关） |

日志字段建议包含：`session_id`、`model_id`、`latency_ms`、`gad7_snapshot_version`，**禁止**记录完整对话于非受控环境（可配置脱敏级别）。

## 4. 打分工具设计（GAD-7 Scoring Engine）

### 4.1 职责划分

| 组件 | 输入 | 输出 |
|------|------|------|
| **NLU 抽取器**（可由自研模型担任） | 最近若干轮对话 + 当前题锚点 | 每项候选分数、置信度、需澄清点 |
| **规则校验器** | 候选分数 | 合法0–3 / 非法则丢弃或标为 `null` |
| **状态聚合器** | 七项分值 | 总分、严重度、`ready_for_summary` |
| **可解释性模块** | 消息 ID + span | `evidence_spans`、`rationale`（短文本） |

### 4.2 单项打分状态机（逻辑）

对每个 `Qi`：

1. **未开始** → 模型/编排按 GAD-7 官方意涵发起自然语言提问（可一次一题或合并澄清，由产品决定）。
2. **已收集语料** → NLU 输出 `score_candidate` 与 `confidence`。
3. **置信度 ≥ τ_high** → 锁定该项分数。
4. **τ_low ≤置信度 < τ_high** → 标记 `needs_clarification`，由 AI 生成**非诱导性**追问（固定追问模板库 + 模型润色）。
5. **置信度 < τ_low** 或语义冲突 → 保持未锁定，继续对话。

阈值 `τ_high`、`τ_low` 为可配置参数（如0.85 / 0.55），并需在离线评测集上标定。

### 4.3 与自研模型的协作模式（推荐）

- **模式 A — 结构化 JSON 侧车**：模型在服务端约束下输出 JSON Schema（`items[].score` 等），经校验后写入快照；自然语言回复与 JSON 分两条通道或同一响应内分段解析。
- **模式 B — 后处理解析**：模型只生成自然语言，独立「抽取模型」或同一模型第二次调用做抽取（成本更高，慎选）。

优先 **模式 A**，便于测试与回归。

### 4.4 离线评测与对齐

- 构建**中文对话 →标准 GAD-7 分数**的标注集（可参考现有 `anotate` 标注规范）。
- 指标：单项准确率、总分 MAE、需澄清轮次、**安全性**（无诱导作答）。
- 每次 `model_id` 升级需跑回归集并记录报告。

### 4.5 合规与安全

- 输出必须包含**不适用场景**与**紧急求助提示**（若用户自述自伤/伤人风险，需走危机话术策略——可由独立分类器触发）。
- 对话与分数访问需**最小权限**与留存策略（GDPR/本地法规按实际适用）。

## 5. 数据模型（后端存储草案）

### 5.1 核心表/集合（概念）

- **sessions**：`session_id`, `user_ref`, `state`, `created_at`, `closed_at`, `model_id`
- **messages**：`message_id`, `session_id`, `role`, `content`, `created_at`
- **gad7_snapshots**：`session_id`, `version`, `items_json`, `total`, `computed_at`（支持历史版本便于审计）

### 5.2 前端所需最小能力

详情接口应足以支撑：**七项进度**、**最新 AI 消息**、**是否可出总结**（`ready_for_summary`）。具体 UI 与视觉由前端另行实现。

---

## 6. 前端对接要点（不含视觉稿）

前端需实现的能力与后端契约对齐即可：

1. **会话创建**与**消息发送**（REST或 SSE）。
2. **展示** `gad7_snapshot`：每项是否已答、是否需要澄清（可用简单图标，设计自定）。
3. **总结页**调用 `GET .../report` 或依赖消息流中最终事件。
4. **错误与重试**：网络失败、会话过期、`503` 模型不可用时的降级文案（由产品给定）。
